# 06 — Security Posture

## 6.1 Threat model (top 6 for SentinelGrid)

1. **Nation-state adversary in the supply chain** — image tampering, dependency attacks
2. **Compromised partner org** uploading malicious "threat intel"
3. **Insider abuse** of platform privileges to leak data
4. **Ransomware / destructive malware**
5. **DDoS** against the public ingestion endpoint
6. **Regional cloud outage** (used as cover for a coordinated attack)

## 6.2 Controls

| Threat | Primary control | Secondary |
|---|---|---|
| Supply chain | cosign-signed images + Kyverno admission verify; SBOM generated by Syft + scanned in CI | Trivy + dependency pin file with hash checking |
| Malicious partner upload | mTLS auth + payload signature verify + per-org rate limits + content sandboxing | ML anomaly scoring before publish |
| Insider abuse | Least-privilege Vault + IAM; dual-control on policy changes; immutable audit trail | UEBA on Vault audit logs in SOC space |
| Ransomware | Object Lock COMPLIANCE on audit + Velero buckets; RDS PITR; short-TTL dynamic creds | Falco runtime detection |
| DDoS | AWS Shield Advanced + WAF + Kong rate-limit + HPA | Anycast edge (CloudFront) + auto-scaling NAT bandwidth |
| Regional outage | Multi-region warm standby + Route53 health-checked failover | Tested semi-annually |

## 6.3 Identity hierarchy

```
Human ops
  └─ AWS SSO ── group claim ── OIDC ──► Vault role ──► (k8s RBAC | IAM)

Service-to-AWS
  └─ K8s ServiceAccount ── projected JWT ── IRSA ──► IAM role (scoped to ARN)

Service-to-service (intra-cluster)
  └─ Vault PKI ── short-lived mTLS cert (24 h) ── Istio sidecar

Service-to-DB
  └─ Vault DB engine ── dynamic Postgres role (1 h) ── RDS IAM auth fallback
```

No long-lived credentials are stored in pods, repos, or CI variables.

## 6.4 Network controls

- VPC private by default; only the ALB sits in public subnets.
- All AWS API calls via VPC endpoints — no traffic leaves the VPC for ECR, S3, KMS, STS, Secrets Manager.
- Kubernetes NetworkPolicy is **deny-by-default** per namespace; explicit allow rules are PR-reviewed.
- Istio mTLS in `STRICT` mode cluster-wide.

## 6.5 Compliance posture

- **FedRAMP High** controls inherited from AWS GovCloud (the case study lives on standard AWS commercial — see `docs/02-infrastructure.md` for the GovCloud migration plan if mandated).
- **NIST 800-53** controls mapped in `docs/compliance/nist-mapping.md`.
- Quarterly penetration test by an external red team; findings tracked in Jira with security-owned SLAs.

## 6.6 Vault specifics

- 3-node Raft HA, KMS auto-unseal (no manual unseal step on pod restart).
- Recovery keys sealed with PGP keys of two officers, stored in physical safes (case-study deliverable: documented procedure only).
- Audit device → STDOUT → Filebeat → Elasticsearch (`sg-audit-*` index, ILM keeps 7 years).
- Mutator API paths (policies, auth methods, mount config) blocked by a network policy at the Vault namespace egress — Terraform via PR is the only path.
