the dashboard Dockerfile was written for the wrong framework. These are now fixed:

| Service | Problem found | Fix applied |
|---------|---------------|-------------|

| `threat-ingestion` | Source existed but Kafka was hardcoded to MSK `SASL_SSL/OAUTHBEARER`; crashed locally. Dockerfile built wheels but **never installed them** (distroless has no pip). | Made Kafka config env-driven (PLAINTEXT local / SASL prod) and resilient if broker down. Fixed Dockerfile to install deps. |
| `threat-analysis` | Empty folder. Dockerfile referenced non-existent `src/consumer.py`. | Created Kafka consumer with MITRE ATT&CK scoring + health/metrics on :8081. Fixed Dockerfile. |
| `intel-distribution` | Empty folder. Dockerfile referenced non-existent Go module. | Created Go STIX 2.1 / TAXII 2.1 publisher (stdlib only) on :8082. |
| `incident-coordination` | Empty folder. Dockerfile referenced non-existent Node build. | Created Node playbook-orchestration service on :8083. |
| `api-gateway` | Empty folder. Dockerfile referenced non-existent `kong.yml` + `opa-policies`. | Created Kong DB-less declarative config + OPA Rego authz policy. |
| (stack) | No way to run the platform locally. | Added `docker-compose.yml` (all services + Redpanda + Prometheus + Grafana) and monitoring provisioning. |

```
POST /v1/events ×6  → threat-ingestion → 202 Accepted
                    → Apache Kafka topic "raw-events" (high-watermark = 6)
                    → threat-analysis consumes, scores via MITRE ATT&CK weights
                    → sg_events_scored_total{verdict="malicious"} 3

                    → consumer group lag = 0  (all messages processed)
```

## 4. Honest caveats (production layers)

These layers are **declarative/config artifacts** intended for a real AWS account and a
live EKS cluster; they are correct as documentation/IaC but are **not exercised** by the
local compose demo, and an earlier audit found issues to fix before a real `terraform apply`:

- **Terraform**: missing `aws_iam_role` for VPC flow logs; EKS module lacks a
  `cluster_security_group_id` output referenced by prod-primary; RDS lacks an `rds_arn`
  output referenced by prod-dr. These will fail `terraform apply` until added.
- **Kubernetes**: image refs use unsubstituted `ACCOUNT_ID`/`IMAGE_TAG` placeholders;
  several services lack ServiceAccount/Service manifests; the kustomization image-tag
  syntax needs correcting; ingress references placeholder ACM/WAF ARNs.
- **ELK / Vault**: configuration is present and coherent but requires the ECK operator /
  a Vault cluster to run — not part of the laptop demo.

The local compose stack is the runnable proof; the AWS layers are the target architecture.
